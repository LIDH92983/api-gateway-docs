Key 认证插件配置参考

功能说明​

Key-Auth 插件实现了基于 API Key 进行认证鉴权的功能，支持从 HTTP 请求的 URL 参数或者请求头解析 API Key，同时验证该 API Key 是否有权限访问。

配置字段​

注意事项​

  • 对于通过认证鉴权的请求，请求的 header 会被添加一个 X-Mse-Consumer 字段，用以标识调用者的名称。

认证配置​

名称数据类型填写要求默认值描述
global_authbool选填（仅场景级别配置）-只能在场景级别配置，若配置为 true，则全局生效认证机制；若配置为 false，则只对做了配置的域名和路由生效。
consumersarray of object必填-配置服务的调用者，用于对请求进行认证。
keysarray of string必填-API Key 的来源字段名称，可以是 URL 参数或者 HTTP 请求头名称。
in_querybooltrue配置为 true 时，网关会尝试从 URL 参数中解析 API Key。
in_headerbooltrue配置为 true 时，网关会尝试从 HTTP 请求头中解析 API Key。

consumers 中每一项的配置字段说明​

名称数据类型填写要求默认值描述
credentialstring必填-配置该 consumer 的访问凭证。
namestring必填-配置该 consumer 的名称。

鉴权配置（非必需）​

名称数据类型填写要求默认值描述
allowarray of string选填（非场景级别配置）-只能在路由或域名等细粒度规则上配置，允许访问的 consumer。

配置级别说明​

在配置插件时，您需要选择规则级别（场景级别、路由级别或域名级别），并根据所选级别输入相应的插件规则。

1. 场景级别配置​

  • 作用范围：适用于整个网关的具体场景，所有该场景下的请求都会受到该配置的影响。
  • 配置内容：
    • global_auth: 该字段用于设置全局认证机制。若配置为 true，则所有请求都必须通过认证；若为 false，则仅对特定的域名和路由生效。
    • consumers: 该字段是一个数组，用于定义所有消费者的认证信息。每个消费者必须提供唯一的凭证和名称，以便于识别和管理。
      • credential: 该字段是消费者的访问凭证，必须是唯一且有效的字符串。
      • name: 该字段是消费者的名称，用于标识和管理不同的消费者。
    • keys: 该字段是一个数组，定义 API Key 的来源字段名称，可以是 URL 参数或 HTTP 请求头的名称。支持多个字段，允许灵活配置 API Key 的获取方式。
global_auth: false
consumers:
  - credential: 2bda943c-ba2b-11ec-ba07-00163e125  # 真实的 clientId
    name: "consumer1"
  - credential: c8c8e9ca-558e-4a2d-bb62-e700dcc40  # 真实的 secretKey
    name: "consumer2"
keys:
  - clientId
  - secretKey

2. 路由级别配置​

  • 作用范围：仅适用于特定的路由，当请求匹配到该路由时，才会应用该配置。
  • 配置内容：可以定义允许访问该路由的消费者，细粒度控制访问权限。
routes:
  - name: "getToken"
    allow:
      - "consumer1"  # 允许的 consumer
    plugins:
      key-auth:
        keys:
          - clientId
          - secretKey

  - name: "protectedResource"
    allow:
      - "consumer1"  # 允许的 consumer
    plugins:
      key-auth:
        keys:
          - clientId
          - token  # 这里的 token 是动态生成的

请求流程​

  1. 请求 getToken 接口：

    • 用户向网关发送请求，携带 clientId 和 secretKey。
    • 请求示例： 
      curl http://your-gateway.com/getToken -H 'clientId: your-client-id' -H 'secretKey: your-secret-key'

  2. 获取 Token：

    • 如果验证通过，网关将请求转发到具体的服务，服务返回一个动态生成的 token。

  3. 请求其他接口：

    • 用户在后续请求中需要携带 clientId 和动态生成的 token。
    • 请求示例： 
      curl http://your-gateway.com/protectedResource -H 'clientId: your-client-id' -H 'Authorization: Bearer your-dynamic-token'

  4. 验证 Token：

    • 网关在接收到请求时，会检查 Authorization 头中的 token 是否有效。
    • 具体的验证流程可能包括：
      • 检查 token 是否存在于数据库或缓存中。
      • 验证 token 的有效期。
      • 检查 token 是否与请求的 clientId 关联。
    • 如果 token 有效，允许访问请求的资源；如果无效，返回 403 错误。

相关错误码​

HTTP 状态码出错信息原因说明
401Request denied by Key Auth check. Invalid clientId or secretKeyclientId 或 secretKey 无效
403Request denied by Key Auth check. Unauthorized tokentoken 无效或未授权

总结​

通过上述配置和流程，您可以实现对 getToken 接口的安全验证，并确保后续请求携带有效的 clientId 和动态生成的 token。请根据实际情况调整配置，以满足您的需求。
