三方接入设计

1. 基础架构和流程设计​

1.1 Client 认证：通过 clientId 和 secretKey 获取 Token​

  1. 客户端发送 clientId 和 secretKey 调用网关配置的 /getToken 接口。
  2. 规则引擎校验 clientId 和 secretKey 的合法性：
    • 验证 clientId：检查 clientId 是否存在于请求中。
    • 验证 secretKey：根据 clientId 从存储中获取对应的 secretKey，并与请求中的 secretKey 进行比较。
  3. 如果 clientId 和 secretKey 验证通过，网关将请求转发到后端服务以生成 Token。
  4. 后端服务返回生成的 Token。
  5. 网关记录 Token 与 clientId 的映射关系，存储在缓存中。

存储方式：

  • 将 Token 与 clientId 映射存入 Redis，设置过期时间与 Token 一致。
  • 示例存储结构： 
    {
      "clientId:abc123": {
        "token": "xyz456",
        "expiresAt": 1699999999
      }
    }

1.2 后续请求校验：Token + clientId 绑定​

客户端在后续请求中必须携带 clientId 和 Token（如通过 Header 或 Query 参数）。
网关将直接根据缓存中的 clientId 和 Token 映射关系进行校验，无需解析 Token。
校验通过后，网关将请求转发至后端服务。

2. 网关认证的详细流程​

2.1 获取 Token 的流程​

Client --> [网关] --> [后端服务] --> [网关] --> Client

客户端向网关发起 POST /getToken 请求，携带：

  • clientId
  • secretKey

网关处理流程：

  1. 验证 clientId 是否存在。
  2. 验证 secretKey 是否与 clientId 绑定：
    • 从存储中获取与 clientId 关联的 secretKey。
    • 比较请求中的 secretKey 和存储中的 secretKey 是否一致。
  3. 如果验证通过，网关将请求转发给后端服务。

后端服务处理流程：

  1. 生成 Token，并返回给网关。
  2. 网关记录映射关系：
    • 在 Redis 或内存中存储 clientId 与 Token 的绑定关系及有效期。
  3. 网关将 Token 返回给客户端。

2.2 访问业务接口的流程​

Client --> [网关] --> 校验通过 --> 转发至后端服务

客户端携带 clientId 和 Token 调用业务接口（如 POST /api/resource）。
网关处理流程：

  1. 检查 clientId 是否存在。
  2. 检查 Token 是否与 clientId 匹配（直接查询 Redis，无需解析 Token）。
  3. 校验通过后，网关将请求转发给后端服务。
  4. 如果校验失败，网关将直接返回 401 错误。

3. 接入配置​

为了使第三方能够成功接入，平台需要进行以下配置：

  1. 注册 Client：

    • 在平台的管理界面中，管理员需要为每个第三方客户端注册 clientId，该 clientId 代表一个场景名称。
    • 确保 clientId 是唯一的， 并且 secretKey 应该是安全生成的。。

  2. 创建资源：

    • 为每个 clientId 创建新的资源(API)。
    • 确保资源的名称和描述清晰，以便于后续管理。

  3. 绑定规则：

    • 为每个资源绑定规则（资源类型为三方调用时建议绑定以下规则，如果不存在新建即可）：
      • 对于 /getToken 资源，配置规则要求请求 Header 中必须携带 secretKey 和 clientId。
      • 对于其他第三方资源，配置规则要求请求中必须携带 clientId 和 Token。

  4. 监控与日志：

    • 配置监控和日志记录，以便跟踪第三方客户端的请求和响应情况。
    • 监控异常请求和潜在的安全威胁。

4. 性能优化点​

  • 避免 Token 解析
    网关通过 Redis 或内存存储 clientId 和 Token 的映射关系，而非直接解析 Token 来校验有效性，从而减少资源消耗。

  • 使用缓存存储 Token
    Redis 是推荐的存储方式，支持高性能查询和分布式部署。
    如果是单节点环境，也可以使用内存存储（如 LRU Cache）。

  • 异步刷新 Token 映射
    在 Token 即将过期时，允许客户端提前调用 /getToken 刷新 Token。
    更新 Redis 的 clientId 和 Token 映射，避免频繁重建会话。

  • 限流和安全性
    在 /getToken 接口中加入限流机制，防止恶意调用。
    对后续业务接口请求进行速率限制（如每秒允许请求数）。

5. 引入规则引擎增强安全性​

通过网关的规则引擎，进一步增强认证和安全机制：

5.1 请求规则​

  • 配置必须携带 clientId 和 Token 的校验规则。
  • 动态校验请求中的 Header 或 Query 参数值是否满足特定格式。

示例规则：

{
  "type": "allow",
  "conditions": [
    {"key": "header.clientId", "operator": "exists"},
    {"key": "header.token", "operator": "exists"},
    {"key": "header.secretKey", "operator": "exists"}  // 确保 secretKey 存在
  ]
}

5.2 自定义逻辑​

  • 使用规则引擎动态配置对请求 Body 或 Header 的校验逻辑，比如：
    • 必须携带特定参数。
    • 参数值必须符合正则表达式。

6. 示例架构图​

Client --> [网关]
               |--> 校验 `clientId` 和 Token 映射 (Redis)
               |--> 转发至后端服务

总结​

通过网关认证机制的改造，您可以实现以下目标：

  • 高效性：网关只校验 Token 的合法性，不解析 Token 内容，避免性能浪费。
  • 安全性：通过规则引擎配置请求校验规则，防止恶意调用。
  • 灵活性：支持动态更新规则，适应不同业务场景。
